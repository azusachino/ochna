# Agent Ergonomics Plan — guide, preflight, findings, semantic search

Four proposals to make ochna self-describing and more useful to coding agents,
plus an evaluation of `sqlite-vec`. Storage stays single-file SQLite — these are
ergonomics and one optional capability, not a second store. (See the standing
decision: SQLite point queries are 2–4 ms even on the 1.25 GB linux index, so no
format change is justified for speed.)

Date: 2026-06-24. Status: workstreams 1 and 2 implemented; workstreams 3 and 4 deferred.

## Framing

An agent that lands in a repo with a `.ochna/` directory has no idea ochna
exists, whether the index is fresh, or how to drive it. Today that knowledge
lives only in `AGENTS.md` of *this* repo. The four workstreams below close the
gap, ordered by leverage and ascending cost:

1. `howto` command teaches usage; `init` drops a thin pointer file (cheap, high
   leverage).
2. A machine-readable preflight verdict so agents gate on freshness (cheap).
3. A repo-local findings log surfaced contextually (medium; optional).
4. `sqlite-vec` semantic search (high value, high cost — pilot only).

## 1. `ochna howto` — the teaching surface (+ a thin `.ochna/` pointer)

An agent (or human) lands in the repo and needs to learn how to drive ochna. Two
discovery paths, one source of truth:

### `ochna howto` — canonical, on-demand
A command that prints how to use ochna. **Preferred over a static doc** because
it's always in sync with the installed binary, is agent-callable, and honors the
global `--json` flag for machine consumption.

- **Human output (`ochna howto`):** the investigate flow —
  `search <name>` → `callers <name>` →
  `node --symbol <name> --include-code`, or `explore <query>` for the combined
  view ("prefer this over recursive grep for symbol lookups") — plus operational
  facts: DB resolves from cwd, no `--workspace` flag, `--json` everywhere,
  `--no-tests` to drop test symbols, and "run `ochna status` first to check
  freshness" (Workstream 2).
- **Machine output (`ochna howto --json`):** a structured capability descriptor
  an agent can consume without scraping prose:
  ```json
  {
    "flow": ["search", "callers", "node", "explore"],
    "commands": { "search": "fuzzy symbol lookup", "callers": "reverse call edges", "...": "..." },
    "globals": ["--json", "--no-tests"],
    "notes": ["db resolves from cwd", "run `ochna status` before trusting results"]
  }
  ```
- **Why a command, not just `--help`:** `--help` lists flags; `howto` teaches the
  *workflow* (which command to reach for, in what order) and is one stable,
  documented thing the guide file and the README both point at.

Implementation: a `Howto` subcommand → `run_howto(json: bool)` in
`src/commands/`. Content is a `const` (human template) plus a `serde` struct
(json). No new dependency.

### `.ochna/AGENT.md` — passive discovery, thin by design
`init`/`sync` (re)write a short tool-owned file so an agent *scanning the repo*
(not yet knowing ochna exists) finds it. To avoid drift it holds **no usage
content** — just provenance + a pointer:

- Header: `<!-- generated by ochna init — do not edit -->`.
- A provenance block filled at generation: indexed SHA, branch, dirty/clean,
  `indexed_at`.
- One line: "This repo is indexed by ochna — run `ochna howto` to learn the
  query flow, `ochna status` to check freshness."

So `howto` is the single source of usage truth; the file is a 5-line signpost
that can't drift because it doesn't duplicate the content. We never touch the
repo's root `AGENTS.md` (human-owned). Implementation: a
`write_agent_pointer(&ochna_dir, &metadata)` call at the tail of `run_init`.

## 2. Preflight verdict — `status --json` with an `ok` gate

Agents should cheaply answer "can I trust this index right now?" before querying.
Fold this into the existing `status` command rather than adding a subcommand.

- **`ochna status --json`** emits a verdict and sets a **non-zero exit code** when
  not ok, so an agent (or shell `&&`) can gate without parsing:
  ```json
  {
    "ok": false,
    "db_present": true,
    "schema": { "expected": 5, "found": 5, "match": true },
    "counts": { "nodes": 41087, "edges": 75102 },
    "freshness": "stale",
    "indexed_sha": "ec4efdb",
    "head_sha": "a1b2c3d",
    "working_tree": "dirty",
    "action": "ochna sync"
  }
  ```
- **Checks, all from data already on hand:**
  - DB present + schema version == binary `SCHEMA_VERSION` (mismatch ⇒ a re-init
    is coming, since migration drops tables — agent should know not to trust
    counts mid-flight).
  - `nodes > 0` sanity.
  - Freshness: compare `project_metadata.git_commit_sha` / `git_status` against a
    live `git rev-parse HEAD` and dirty check. `fresh` if SHA matches and clean;
    `stale` otherwise; `unknown` outside a git repo.
- **`action`** is the single next command to make it ok (`ochna init` / `sync`).

Cheap, deterministic, no new tables. This is the "quick check before really
using ochna" the agent guide points at.

## 3. Repo-local findings log (optional)

Let agents persist critical findings during investigation ("`X` is the real
entry point", "`Y` resolves dynamically so `callers` undercounts") and have them
**resurface contextually** in later queries.

- **Store:** `.ochna/findings.jsonl` — append-only, greppable, agent-friendly.
  **Deliberately outside `ochna.db`**: the graph DB is re-derivable and gets
  *dropped* on schema migration; findings are human/agent knowledge that must
  survive a reindex. One record per line:
  ```json
  {"ts":"2026-06-24T10:00:00Z","symbol":"mutex_lock","file":"kernel/locking/mutex.c","note":"fastpath is inlined; callers undercounts","by":"agent"}
  ```
- **Commands:**
  - `ochna note add --symbol <name> [--file <path>] "<text>"`
  - `ochna note list [--symbol <name>] [--json]`
- **Payoff — contextual resurfacing:** when `node`/`search` returns a symbol that
  has findings, append a `"findings": [...]` field (JSON) or a `note:` line
  (human). That closes the loop: a finding saved once shows up automatically the
  next time anyone touches that symbol.
- **Scope guard / relation to asobi:** keep this *minimal and repo-local* —
  symbol-anchored provenance, not a knowledge graph. Cross-session, cross-repo
  decisions belong in asobi's decision log; the guide should say so to avoid
  reinventing it. Mark this workstream optional — ship 1 & 2 first and see if the
  need is real before adding write paths.

## 4. `sqlite-vec` evaluation — semantic symbol search

### What it brings
FTS5 is lexical only. `sqlite-vec` adds a `vec0` virtual table for vector KNN, in
the **same `.ochna/ochna.db` file** (one C extension, in-process) — consistent
with the single-store principle. New capabilities:

- **Natural-language symbol search:** "find the function that throttles retries"
  → embed name + signature + doc_comment, KNN against the query embedding.
- **"More like this":** given a symbol, find structurally/semantically similar
  ones — useful for an agent mapping an unfamiliar subsystem.

### Query & performance
- Stable `sqlite-vec` does **brute-force KNN** (full scan per query; no ANN index
  yet). Latency ≈ O(N · dim):
  - netty (41k nodes), 384-dim: a few ms — fine.
  - linux (1.4M nodes): a full scan is ~100 ms–seconds — too slow for the
    interactive 2–4 ms budget unless N is cut.
  - **Mitigation:** FTS5 prefilter → vec rerank (lexical narrows to hundreds,
    vec reorders), or only embed a subset (exported/public API symbols).
- **Storage:** 384-dim float32 = 1536 B/vector. Embedding *every* node on linux
  ≈ 1.4M × 1.5 KB ≈ **2.1 GB — larger than the entire current DB.**
  - **Mitigation:** int8 quantization (→ ~540 MB) and/or embed only public
    symbols, not every local.

### Downsides (the real cost)
- **Embedding generation breaks the "pure-Rust, offline, tree-sitter-only"
  simplicity.** You need a model: bundle ONNX/candle (heavy build, large binary)
  or call an external API (network dependency, non-offline, per-token cost). This
  is the dominant cost, not the vector storage.
- **Build complexity:** loading the `vec0` extension via rusqlite
  (`loadable_extension`) plus shipping the native lib cross-platform.
- **Brute-force scaling** on the giants (above).
- **Vector bloat** can exceed the graph itself (above).
- **Non-determinism:** embeddings are fuzzy; agents often want the deterministic
  structural answer ochna already gives. Semantic search is additive discovery,
  not a replacement for `callers`/`node`.

### Recommendation
Genuinely valuable and agent-friendly, but **highest cost and risk — do not put
it in core.** Gate behind a `--features semantic` build flag. Embed only a
curated subset (exported/public API symbols), quantize to int8, and prefilter
with FTS5 before the vec rerank. **Pilot on netty/tokio first**; only consider the
giants after the embedding-source and storage questions are settled. Evaluate it
as a new *feature* on its own merits, never as a performance fix.

## Sequencing

| # | Workstream | Cost | Dep | Priority |
|---|------------|------|-----|----------|
| 1 | `ochna howto` + `.ochna/AGENT.md` pointer | low | none | do first |
| 2 | `status --json` preflight verdict | low | none | do first |
| 3 | findings log + resurfacing | med | 1 (guide points to it) | optional / after 1–2 |
| 4 | `sqlite-vec` semantic search | high | feature flag | pilot only, separate track |

1 and 2 are a small, self-contained PR with no new dependencies and immediate
agent payoff. 3 is opt-in. 4 is a research spike, not a core commitment.

## Open questions

- Pointer file: regenerate on every `sync`, or only `init` + when stale?
  (Leaning: every run — it's cheap and keeps the provenance block live.)
- Preflight: is a non-zero exit code on `stale` too aggressive for interactive
  human use? Maybe gate the exit code behind `--strict`.
- Findings: do we dedupe on `(symbol, note)`, and cap file size like asobi's
  50-obs rule?
- sqlite-vec: where do embeddings come from — bundled local model vs. API — is the
  blocking decision before any of workstream 4 starts.
